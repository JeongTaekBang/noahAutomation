# PO 매입대사 — 한눈에 이해하기

> 운영자 관점 시각 가이드. 자세한 내부 로직은 [PO_RECONCILIATION.md](PO_RECONCILIATION.md) 참고.

## 한 줄 요약

**회계팀이 AX에 입력한 GRN 금액**과 **공장 출고 + 내부 PO 금액**이 AX PO 단위로 맞는지 매월 점검하고, 안 맞는 건은 대응 액션을 도출한다.

---

## 큰 그림

```
   ┌────────────────────┐    ┌──────────────────────┐    ┌────────────────┐
   │  ① 회계 GRN         │    │  ② 공장 출고 리스트   │    │ ③ 내부 PO       │
   │  (AX 매입처리 결과)  │    │  (RCK 출고 실적)      │    │ (담당자 관리)   │
   │  Noah purchase_GRN  │    │  2026리스트_RCK      │    │ NOAH_SO_PO_DN │
   └──────────┬─────────┘    └──────────┬───────────┘    └────────┬───────┘
              │                          │                         │
              │                          │  ND/NO → AX PO 매핑      │
              │                          │  P###### → 그대로        │
              │                          ▼                         │
              │              ┌──────────────────────┐              │
              └─────────────▶│   AX PO (P######)    │◀─────────────┘
                             │   = 공통 조인 키      │
                             └──────────┬───────────┘
                                        ▼
                          ┌──────────────────────────┐
                          │   AX PO 단위 금액 비교    │
                          │   GRN ↔ PO ↔ 출고         │
                          └──────────┬───────────────┘
                                     ▼
                          ┌──────────────────────────┐
                          │  대사결과_PXX.xlsx        │
                          │  (5~6개 시트 + 합계)      │
                          └──────────────────────────┘
```

핵심 한 줄: **GRN이 기준**이고, PO와 출고가 거기 맞춰 검증된다. PO/출고에는 있는데 GRN에 없으면 → 회계팀이 GRN 처리해야 할 후보.

## 데이터 소스 3개

| | 소스 | 파일 | 키 컬럼 | 금액 컬럼 | 누가 만드나 |
|--|------|-----|--------|---------|------------|
| ① | **회계 GRN** | `po_reconciliation/2026/Pxx/Noah purchase_GRN_Pxx_2026.xlsx` | `Purchase order` | `Cost amount physical` | 회계팀 |
| ② | **공장 출고** | `po_reconciliation/2026/Pxx/2026리스트_RCK_Pxx.xlsx` (Delivery 시트) | `RCK ODER` | `계산서금액` | 공장 |
| ③ | **내부 PO** | `NOAH_SO_PO_DN.xlsx` (PO_국내 + PO_해외) | `PO_ID` ↔ `AX PO` | `Total ICO` | 사용자 본인 |

폴더 구조: `po_reconciliation/{연도}/{월코드}/` — 연도 폴더가 없는 평면 구조도 자동 호환.

## AX PO 키 통일

세 파일이 서로 다른 키를 쓰기 때문에 **AX PO(P######)** 로 통일한다.

```
회계 GRN          : Purchase order (P022562)         그대로 사용
공장 출고         : RCK ODER                         두 가지 패턴
                    ├─ ND-xxxx / NO-xxxx           → PO시트에서 AX PO 매핑
                    └─ P######                     → 그대로 사용
내부 PO            : AX PO (P022562)                  그대로 사용
                    PO_ID(ND/NO)는 매핑용
```

**1:N 매핑** — 한 PO_ID가 여러 AX PO를 가질 수 있음:

```
ND-0214 ──┬── P023006 (Total ICO: 176,000)
          └── P023007 (Total ICO: 479,235)

출고리스트:  ND-0214 한 줄 (계산서금액 655,235)
GRN:        P023006, P023007 별도 행
→ 비교는 AX PO 단위로 (출고 행이 자동 복제됨)
```

## 매칭 판정 (8가지 상태)

GRN의 각 AX PO에 대해 PO/출고 존재 여부 + 금액 비교로 결정:

```
GRN에 있는 AX PO                                         매칭상태
  ├─ 내부 PO Invoiced에 있음                              → 일치 / 불일치
  │     └ 해외 PO면                                      → 일치(해외) / 불일치(해외)
  ├─ 내부 PO엔 없고 출고에 있음
  │     └ Type = Service/YTC                             → 일치(서비스) / 불일치(서비스)
  │     └ Type = Product                                 → AX PO 누락 (PO시트에 AX PO 비어있음)
  └─ 둘 다 없음                                           → 서비스/기타

GRN에 없는 AX PO (PO Invoiced 또는 출고에만 존재)        → GRN 미포함 (별도 시트)
```

## 결과 파일 — 대사결과_PXX.xlsx

### 시트 구성

| 시트 | 무엇이 들어있나 |
|------|---------------|
| **대사** | AX PO별 1행 요약 + 합계 블록(아래) |
| **GRN_미포함** | GRN에 없는 AX PO 목록 — **AX에 GRN 처리 필요한 건** |
| raw_data | 소스별(GRN/출고/PO) 상세 long-format |
| raw_data_미포함 | GRN 미포함 건의 소스별 상세 |
| AX_PO_미입력 | Invoiced인데 PO시트에 AX PO 비어있는 건 |
| 범례 | 매칭상태 설명표 |

### 대사 시트 — AX PO별 1행

```
AX PO    | 비교_금액  | GRN_금액   | 매칭상태
P022562  |   400,800 |   400,800 | 일치
P022939  |   160,000 |   160,000 | 일치(서비스)
P023006  |   176,000 |   176,000 | 일치
...
P023891  |   598,500 |   598,500 | 일치
```

- **비교_금액** = PO_금액 우선, 없으면 출고_금액
- **GRN_금액** = 회계 GRN 측 금액
- **매칭상태** = 위 8가지 중 하나

### 합계 블록 (대사 시트 표 아래, 테이블 외부)

3개 섹션으로 구성된 검증용 요약:

```
┌─ GRN 측 ─────────────────────────────────────────────┐
│ GRN 합계         │ 851,330,902  │ 전체 GRN raw 합계  │
│ 비교가능 합계    │ 851,330,902  │ PO/출고 매칭        │
│ 비교불가         │           0  │ 서비스/기타         │
└───────────────────────────────────────────────────────┘

┌─ AX에 GRN 처리 필요 ────────────────────────────────────┐
│ GRN 미포함 합계  │ 586,433,214  │ 별도 시트로 안내      │
│   Product       │ 553,615,048  │ 내부 PO 측 (관리 대상)│
│   Service       │  32,818,166  │ 출고만 (PO시트 외)    │
│ AX PO 미입력     │           0  │ Invoiced인데 AX PO ⊘ │
└────────────────────────────────────────────────────────┘

┌─ PO Invoiced 검증 ──────────────────────────────────────┐
│ PO Invoiced 합계 │ 1,379,826,318│ 당월 전체 (302건)    │
│   국내           │   380,731,017│ 171건 / 773수량      │
│   해외           │   999,095,301│ 131건 / 3,021수량    │
└─────────────────────────────────────────────────────────┘
```

### GRN_미포함 시트

```
AX PO    | Type    | 매입금액
P023344  | Product | 278,627,389  ← 큰 건부터 정렬
P023252  | Product | 123,828,484
...
P023547  | Service |     342,400  ← 출고만 있고 PO Invoiced에는 없음
```

- **Type=Product** → 내부 PO 관리 대상. PO Invoiced 측 미매칭. 가장 먼저 회계팀에 GRN 요청.
- **Type=Service** → PO시트 외 항목 (서비스 계약, Confirmed 상태 PO 등).
- **매입금액** = AX에 GRN으로 처리할 금액 (PO 우선·출고 fallback).

## 자주 보는 케이스 + 대응 액션

| 매칭상태 | 무엇을 의미하나 | 액션 |
|---------|---------------|------|
| **일치 / 일치(해외)** | PO·GRN 금액 일치 | 정상 — 추가 작업 없음 |
| **불일치 / 불일치(해외)** | PO·GRN 금액 차이 | 한쪽 또는 양쪽 금액 확인. 환율, 단가, 수량, BOM 변경 등 추적 |
| **일치(서비스)** | 서비스/YTC, 출고·GRN 금액 일치 | 정상 |
| **불일치(서비스)** | 서비스 출고 ≠ GRN | 출고 또는 GRN 금액 확인 |
| **AX PO 누락** | 출고했고 GRN도 됐는데 PO시트에 AX PO 빈 칸 | NOAH_SO_PO_DN의 해당 PO에 AX PO 입력 — 다음 달부터 정상 매칭 |
| **서비스/기타** | GRN에만 있고 출고·PO 모두 없음 | 회계팀과 정합성 확인 (서비스 계약, 일회성 매입 등) |
| **GRN 미포함 (Product)** | PO Invoiced·출고는 됐는데 GRN 안 됨 | **회계팀에 GRN 처리 요청** |
| **GRN 미포함 (Service)** | 출고만 됐고 PO Invoiced·GRN 둘 다 없음 | PO 상태 확인 (Confirmed로 머물러 있는지 등) |

## 검증 식 — 합계가 맞는지 한눈에

```
PO Invoiced 합계
  = (대사 시트의 PO 측 매칭금액)
  + (GRN 미포함 Product 합계)
  + (AX PO 미입력 합계)

예 P04: 1,379,826,318
  = 826,211,270 (대사 PO 측)
  + 553,615,048 (GRN 미포함 Product)
  + 0           (AX PO 미입력)
```

세 값이 PO Invoiced 합계와 일치하면 모든 PO가 어딘가에 잡혔다는 뜻. 안 맞으면 데이터 누락이나 필터 문제.

## 월별 운영 절차

```
1. 회계팀이 Pxx GRN 파일 전달
2. po_reconciliation/{YYYY}/Pxx/ 에 입력 파일 3개 배치
   ├─ Noah purchase_GRN_Pxx_*.xlsx     (회계 제공)
   ├─ 2026리스트_RCK_Pxx.xlsx          (공장 출고)
   └─ NOAH_SO_PO_DN.xlsx              (담당자 본인 관리)
3. python reconcile_po.py Pxx     ← 또는 bat 메뉴 [R]
4. 대사결과_Pxx.xlsx 열고 합계 블록 확인
5. 불일치/AX PO 누락/GRN 미포함 건 처리
   ├─ 불일치 → 원인 추적
   ├─ AX PO 누락 → PO시트에 AX PO 입력
   └─ GRN 미포함 Product → 회계팀에 GRN 처리 요청
6. 다음 달 재실행 시 변동 추적
```
