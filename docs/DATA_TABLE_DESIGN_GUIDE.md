# 부서별 데이터 테이블 설계 가이드

## ERP를 이해하는 가장 빠른 방법

**ERP 없이 직접 만들어 보는 것입니다.**

ERP는 거창해 보이지만, 결국 이 문서에 있는 테이블들의 집합체입니다. D365든 SAP든, 안을 열어보면 수주 테이블, 발주 테이블, 출고 테이블, 계정과목... 전부 같은 구조입니다.

Excel로 수주 대장, 발주 대장, 출고 대장을 직접 설계하고 운영해 보면:

- **"왜 헤더와 라인을 분리하는가"** — 주문 1건에 품목 10개를 한 행에 넣으면 관리가 안 됨 (1:N 관계)
- **"왜 Status 컬럼이 필요한가"** — Open→Sent→Confirmed 같은 상태 흐름이 없으면 진행 현황 파악 불가
- **"왜 마스터 데이터를 따로 두는가"** — 고객명을 매번 수주마다 입력하면 오타·불일치 발생
- **"왜 Key가 중요한가"** — SO↔PO를 Line item으로 조인했다가 본체+부속 합산 케이스에서 깨지는 경험

이걸 몸으로 겪고 나서 ERP를 보면 **"아, 이게 이 테이블이구나"**가 바로 보입니다.

NOAH 프로젝트가 이 과정 자체입니다:
1. Excel에 SO/PO/DN 시트를 직접 설계 → ERP의 Sales/Purchase/Logistics 모듈 이해
2. 테이블 간 Key로 조인 → ERP의 관계형 DB 이해
3. SQLite + 대시보드 → ERP의 BI/리포팅 이해

**ERP 교육을 100시간 받는 것보다, Excel로 한 번 만들어 보는 게 훨씬 빠릅니다.**

이 문서는 그 역설계 가이드입니다. 각 부서별로 "어떤 테이블이 필요하고, Excel로 어떻게 시작하며, ERP에서는 어떤 테이블에 해당하는지"를 정리합니다.

> **참고**: SQL 예시는 개념 이해용입니다. 실제 운영 쿼리는 테이블/컬럼명, 조인 키, 집계 레벨을 환경에 맞게 조정해야 합니다. 실제 NOAH 프로젝트 쿼리는 `sql/order_book.sql`과 `dashboard.py`를 참조하세요.

## 설계 원칙

1. **의사결정에서 역추적** — "이 사람이 매일 봐야 하는 숫자가 뭔가?"에서 시작
2. **헤더/라인 분리** — 주문 1건(헤더) + 품목 N건(라인) 구조가 기본
3. **마스터/트랜잭션 분리** — 고객 정보(마스터)는 한 곳에, 거래(트랜잭션)에서 참조
4. **이벤트 기록** — 상태 변경은 "현재 상태"만 저장하지 말고 이력(날짜+상태)도 남기기

## 1. 영업 (Sales)

### 의사결정
- 이번 달 수주 얼마? 목표 대비?
- 어떤 고객이 얼마나 사는가?
- 납기에 맞출 수 있는가?
- 견적 대비 수주 전환율?

### 필요 테이블

```
[고객 마스터] ──1:N──→ [수주 헤더] ──1:N──→ [수주 라인]
                            │
                            └──1:N──→ [견적 헤더] ──1:N──→ [견적 라인]
```

| 테이블 | 주요 컬럼 | PK |
|--------|----------|-----|
| **고객 마스터** | 고객ID, 고객명, 사업자번호, 업종, 섹터, 담당 영업 | 고객ID |
| **견적 헤더** | 견적ID, 고객ID, 견적일, 유효기간, 상태, 금액합계 | 견적ID |
| **견적 라인** | 견적ID, 라인번호, 품목, 수량, 단가, 금액 | 견적ID + 라인번호 |
| **수주 헤더** | SO_ID, 고객ID, 수주일, 납기일, 상태, 금액합계 | SO_ID |
| **수주 라인** | SO_ID, 라인번호, 품목, 수량, 단가, 금액, 납기일 | SO_ID + 라인번호 |

### 대시보드 지표
- 월별 수주 금액/건수 추이
- 고객별/섹터별 매출 분석
- 견적→수주 전환율
- 납기 준수율

### ERP 없이 Excel로 시작하기

**시트 1: 고객 마스터**

| 고객ID | 고객명 | 사업자번호 | 섹터 | 담당 영업 |
|--------|--------|----------|------|----------|
| C-001 | (주)삼신 | 123-45-67890 | Oil&Gas | 김OO |
| C-002 | 오토밸브 | 234-56-78901 | Water | 이OO |

**시트 2: 수주 대장**

| SO_ID | 고객ID | 수주일 | 라인 | 품목 | 수량 | 단가 | 금액 | 납기일 | 상태 |
|-------|--------|-------|------|------|------|------|------|--------|------|
| SOD-001 | C-001 | 2026-01-10 | 1 | NA015 | 10 | 500,000 | 5,000,000 | 2026-03-01 | 미출고 |
| SOD-001 | C-001 | 2026-01-10 | 2 | 부싱 | 10 | 50,000 | 500,000 | 2026-03-01 | 미출고 |

**시트 3: 견적 대장** (선택)

| 견적ID | 고객ID | 견적일 | 품목 | 수량 | 금액 | 상태 | SO_ID |
|--------|--------|-------|------|------|------|------|-------|
| QT-001 | C-001 | 2026-01-05 | NA015 | 10 | 5,500,000 | 수주 | SOD-001 |
| QT-002 | C-002 | 2026-01-08 | IQ3 | 5 | 3,000,000 | 실주 | |

> **NOAH 프로젝트가 바로 이 구조입니다.** `NOAH_SO_PO_DN.xlsx`의 SO_국내/SO_해외 시트가 수주 대장이고, 이걸 sync_db.py로 SQLite에 넣어 대시보드를 만들었습니다.

```sql
-- Excel만으로 가능한 분석: 월별 수주 추이
SELECT SUBSTR(수주일, 1, 7) AS 월, SUM(금액) AS 수주금액
FROM 수주대장 GROUP BY SUBSTR(수주일, 1, 7)

-- 고객별 매출 Top 10
SELECT 고객ID, SUM(금액) AS 매출 FROM 수주대장
GROUP BY 고객ID ORDER BY 매출 DESC LIMIT 10

-- 견적→수주 전환율
SELECT COUNT(CASE WHEN 상태='수주' THEN 1 END) * 100.0 / COUNT(*) AS 전환율
FROM 견적대장
```

## 2. 구매 (Procurement)

### 의사결정
- 수주 대비 발주를 다 했는가?
- 공장 납기를 맞출 수 있는가?
- 원가(ICO)가 적정한가?

### 필요 테이블

```
[공급자 마스터] ──1:N──→ [발주 헤더] ──1:N──→ [발주 라인]
                              │
                   [수주 라인] ←── SO_ID로 연결
```

| 테이블 | 주요 컬럼 | PK |
|--------|----------|-----|
| **공급자 마스터** | 공급자ID, 공급자명, 연락처, 리드타임 | 공급자ID |
| **발주 헤더** | PO_ID, SO_ID, 공급자ID, 발주일, 상태 | PO_ID |
| **발주 라인** | PO_ID, 라인번호, 품목, 수량, 단가, 금액, 납기 | PO_ID + 라인번호 |

### 주의: SO↔PO 관계
- 1:1이 아닐 수 있음 (SO 여러 라인 → PO 1라인 합산 발주)
- SO_ID로만 연결하고, 커버리지는 PO Status 기반 판정 권장
- Status 흐름: Open → Sent → Confirmed → Invoiced

### 대시보드 지표
- 발주 커버리지 (미발주/부분발주/발주확정)
- 공급자별 리드타임
- 원가 마진 분석

### ERP 없이 Excel로 시작하기

**시트: 발주 대장**

| PO_ID | SO_ID | 공급자 | 발주일 | 라인 | 품목 | 수량 | 단가(ICO) | 금액 | 납기 | 상태 |
|-------|-------|--------|-------|------|------|------|----------|------|------|------|
| ND-001 | SOD-001 | NOAH공장 | 2026-01-15 | 1 | NA015+부싱 | 10 | 400,000 | 4,000,000 | 2026-02-20 | Confirmed |
| ND-002 | SOD-002 | NOAH공장 | 2026-01-20 | 1 | IQ3 | 5 | 600,000 | 3,000,000 | 2026-03-10 | Open |

> **NOAH의 PO_국내/PO_해외 시트가 이 구조입니다.** Status 흐름(Open→Sent→Confirmed→Invoiced)만 잘 관리하면 발주 커버리지 대시보드가 바로 나옵니다.

```sql
-- 발주 커버리지: SO별 PO 존재 + 상태 확인
-- 주의: 1:N 관계이므로 PO를 먼저 SO_ID 단위로 집계 후 조인
WITH po_agg AS (
    SELECT SO_ID,
           GROUP_CONCAT(DISTINCT 상태) AS po_상태목록
    FROM 발주대장
    WHERE 상태 != 'Cancelled'
    GROUP BY SO_ID
)
SELECT s.SO_ID,
       CASE WHEN p.SO_ID IS NULL THEN '미발주'
            WHEN p.po_상태목록 LIKE '%Open%'
             AND p.po_상태목록 LIKE '%Confirmed%' THEN '부분발주'
            WHEN p.po_상태목록 = 'Open' THEN '미발주'
            ELSE '발주확정' END AS 커버리지
FROM (SELECT DISTINCT SO_ID FROM 수주대장) s
LEFT JOIN po_agg p ON s.SO_ID = p.SO_ID

-- 마진 분석: 양쪽 먼저 집계 후 조인 (뻥튀기 방지)
SELECT s.SO_ID, s.매출, p.원가, s.매출 - p.원가 AS 마진
FROM (SELECT SO_ID, SUM(금액) AS 매출 FROM 수주대장 GROUP BY SO_ID) s
JOIN (SELECT SO_ID, SUM(금액) AS 원가 FROM 발주대장 GROUP BY SO_ID) p
  ON s.SO_ID = p.SO_ID
```

## 3. 물류/출하 (Logistics)

### 의사결정
- 오늘/이번 주 출고할 건?
- 선적 진행 상황?
- 운송 업체 배정?

### 필요 테이블

```
[수주 라인] ──1:N──→ [출고 헤더] ──1:N──→ [출고 라인]
                          │
                          └──→ [선적 정보] (해외만)
```

| 테이블 | 주요 컬럼 | PK |
|--------|----------|-----|
| **출고 헤더** | DN_ID, SO_ID, 출고일, 상태 | DN_ID |
| **출고 라인** | DN_ID, SO_ID, 라인번호, 품목, 수량, 금액 | DN_ID + SO_ID + 라인번호 |
| **선적 정보** | DN_ID, 공장출고일, 픽업일, 선적예정일, 선적일, B/L, 운송업체 | DN_ID |

### 대시보드 지표
- 일별/월별 출고 현황
- 선적 파이프라인 (출고→픽업→선적)
- 리드타임 (수주→출고, 출고→픽업→선적)
- 납기 준수율

### ERP 없이 Excel로 시작하기

**시트 1: 출고 대장 (국내)**

| DN_ID | SO_ID | 고객명 | 품목 | 수량 | 금액 | 출고일 | 세금계산서 발행일 |
|-------|-------|--------|------|------|------|--------|--------------|
| DND-001 | SOD-001 | (주)삼신 | NA015 | 10 | 5,000,000 | 2026-02-25 | 2026-02-28 |

**시트 2: 출고 대장 (해외) — 선적 정보 포함**

| DN_ID | SO_ID | 고객명 | 품목 | 수량 | 금액 | 출고일 | 픽업일 | 선적예정일 | 선적일 | B/L | 운송업체 |
|-------|-------|--------|------|------|------|--------|-------|----------|-------|-----|---------|
| DNO-001 | SOO-001 | ABC Corp | IQ3 | 50 | 30,000,000 | 2026-02-20 | 2026-02-22 | 2026-03-01 | 2026-03-03 | BL123 | DHL |

> **NOAH의 DN_국내/DN_해외 시트가 이 구조입니다.** 국내는 세금계산서 발행일까지, 해외는 선적 파이프라인(출고→픽업→선적) 전체를 한 시트에서 관리합니다.

```sql
-- Order Book (잔고) = 수주 - 출고 (양쪽 먼저 집계 후 조인)
SELECT so.SO_ID, so.수주금액 - COALESCE(dn.출고금액, 0) AS 잔고
FROM (SELECT SO_ID, SUM(금액) AS 수주금액 FROM 수주대장 GROUP BY SO_ID) so
LEFT JOIN (SELECT SO_ID, SUM(금액) AS 출고금액 FROM 출고대장 GROUP BY SO_ID) dn
  ON so.SO_ID = dn.SO_ID
WHERE so.수주금액 - COALESCE(dn.출고금액, 0) > 0

-- 해외 물류 리드타임
SELECT DN_ID,
       JULIANDAY(픽업일) - JULIANDAY(출고일) AS 출고_픽업,
       JULIANDAY(선적일) - JULIANDAY(픽업일) AS 픽업_선적
FROM 출고대장_해외 WHERE 선적일 IS NOT NULL

-- 세금계산서 미발행 (컬럼명에 공백 포함 시 [] 사용)
SELECT * FROM 출고대장_국내
WHERE 출고일 IS NOT NULL
  AND ([세금계산서 발행일] IS NULL OR TRIM([세금계산서 발행일]) = '')
  AND 금액 > 0
```

## 4. 생산 (Production)

### 의사결정
- 오늘/이번 주 생산할 건? 우선순위?
- 생산 능력(Capacity) 대비 부하?
- 자재가 확보되어 있는가?
- 불량률/재작업률?
- 공정별 리드타임?

### 필요 테이블

```
[발주 라인] ──→ [생산 오더] ──1:N──→ [공정 라우팅]
                    │                     │
                    │                [작업 실적] (공정별 시작/완료/수량)
                    │
                    ├──1:N──→ [자재 소요(BOM)]
                    │              │
                    │         [자재 불출] (실제 투입)
                    │
                    └──→ [품질 검사]
```

| 테이블 | 주요 컬럼 | PK |
|--------|----------|-----|
| **생산 오더** | 생산오더ID, PO_ID/SO_ID, 품목, 계획수량, 완료수량, 상태, 계획시작일, 계획완료일, 실제시작일, 실제완료일 | 생산오더ID |
| **공정 라우팅** | 생산오더ID, 공정번호, 공정명, 작업장, 계획시간, 실제시간 | 생산오더ID + 공정번호 |
| **작업 실적** | 생산오더ID, 공정번호, 작업일, 양품수량, 불량수량, 작업시간, 작업자 | 생산오더ID + 공정번호 + 작업일 |
| **BOM (자재 소요)** | 품목코드, 자재코드, 소요수량, 단위 | 품목코드 + 자재코드 |
| **자재 불출** | 생산오더ID, 자재코드, 불출수량, 불출일 | 생산오더ID + 자재코드 + 불출일 |
| **품질 검사** | 검사ID, 생산오더ID, 검사일, 검사수량, 합격수량, 불량수량, 불량유형 | 검사ID |

### 주요 관계
- 생산 오더 ← PO_ID 또는 SO_ID로 수주/발주와 연결
- BOM은 품목 마스터 기준 (생산 오더별 아님), 생산 오더에서 참조
- 작업 실적은 공정별로 쌓임 — 공정 리드타임 분석의 소스

### 대시보드 지표
- 생산 계획 vs 실적 (일별/주별)
- Capacity 가동률 (계획시간 / 가용시간)
- 공정별 리드타임 (계획 vs 실제)
- 불량률 추이 (불량수량 / 검사수량)
- 자재 가용성 (BOM 소요 vs 재고)
- 납기 준수율 (계획완료일 vs 실제완료일)

### ERP 없이 Excel로 시작하기

공정 라우팅과 작업 실적은 Excel 시트 하나로 관리 가능합니다.
현재 NOAH 프로젝트(SO/PO/DN → SQLite → 대시보드)와 동일한 패턴입니다.

**시트 1: 공정 마스터** — 제품별 표준 공정 정의

| 제품 | 공정번호 | 공정명 | 작업장 | 표준시간(h) |
|------|---------|--------|-------|-----------|
| NA015 | 10 | 기계가공 | MC-01 | 2.0 |
| NA015 | 20 | 조립 | ASM-01 | 3.0 |
| NA015 | 30 | 배선 | WIRE-01 | 1.0 |
| NA015 | 40 | 검사 | QC-01 | 0.5 |

생산 대시보드

**분석 예시:**

```sql
-- 공정별 평균 리드타임 (계획 vs 실제)
SELECT 공정명,
       AVG(계획시간) AS 평균_계획,
       AVG(실제시간) AS 평균_실제,
       AVG(실제시간) - AVG(계획시간) AS 차이
FROM 작업실적
GROUP BY 공정명

-- 공정별 불량률
SELECT 공정명,
       SUM(양품 + 불량) AS 총수량,
       SUM(불량) AS 불량수량,
       ROUND(SUM(불량) * 100.0 / SUM(양품 + 불량), 1) AS 불량률
FROM 작업실적
GROUP BY 공정명

-- 작업장별 가동률 (월별)
SELECT 작업장, SUBSTR(작업일, 1, 7) AS 월,
       SUM(실제시간) AS 실제가동,
       COUNT(DISTINCT 작업일) * 8 AS 가용시간,  -- 1일 8시간 기준
       ROUND(SUM(실제시간) * 100.0 / (COUNT(DISTINCT 작업일) * 8), 1) AS 가동률
FROM 작업실적
JOIN 공정마스터 USING (제품, 공정번호)
GROUP BY 작업장, SUBSTR(작업일, 1, 7)
```

|------|---------|--------|------|--------|------|

> TB만 있으면 P&L/BS가 나옵니다. 미수금 대장은 출고 대장에서 파생 가능하지만, 수금 정보가 추가되어야 Aging 분석이 됩니다.

```sql
-- P&L (월별) — Trial_Balance + 계정과목_마스터 조인
SELECT tb.기간,
       SUM(CASE WHEN m.대분류='매출' THEN tb.대변 - tb.차변 END) AS 매출,
       SUM(CASE WHEN m.대분류='원가' THEN tb.차변 - tb.대변 END) AS 원가,
       SUM(CASE WHEN m.대분류='판관비' THEN tb.차변 - tb.대변 END) AS 판관비
FROM Trial_Balance tb
JOIN 계정과목_마스터 m ON tb.계정코드 = m.계정코드
WHERE m.[BS/PL] = 'PL'
GROUP BY tb.기간

-- AR Aging — 미수금 대장에서 경과일 계산
SELECT 고객,
       SUM(CASE WHEN 경과일 <= 30 THEN 잔액 END) AS "30일이내",
       SUM(CASE WHEN 경과일 BETWEEN 31 AND 60 THEN 잔액 END) AS "31-60일",
       SUM(CASE WHEN 경과일 BETWEEN 61 AND 90 THEN 잔액 END) AS "61-90일",
       SUM(CASE WHEN 경과일 > 90 THEN 잔액 END) AS "90일초과"
FROM 미수금_대장
WHERE 잔액 > 0
GROUP BY 고객
```

## 6. 경영/관리 (Management)

### 의사결정
- Order Book (Backlog) 건전성
- Book-to-Bill 추이
- 부서별/제품별 수익성
- 인력 대비 매출 효율

### 필요 테이블

경영은 별도 테이블보다 **위 부서 테이블의 조합**에서 나옵니다:

```
Order Book = 수주 - 출고 (잔고)
마진       = 수주 금액 - 발주 원가
BtB        = 월별 수주 / 월별 출고
```

| 지표 | 소스 테이블 | 계산 |
|------|-----------|------|
| **Order Book** | 수주 + 출고 | 수주금액 - 출고금액 = 잔고 |
| **Backlog Cover** | Order Book + 출고 | 잔고 / 최근3개월 평균출고 |
| **Book-to-Bill** | 수주 + 출고 | 월별 수주 / 월별 출고 |
| **마진율** | 수주 + 발주 | (매출 - ICO) / 매출 |
| **납기 준수율** | 수주 + 출고 | 약속납기 내 출고 비율 |
| **전환율** | 견적→수주→발주→출고 | 각 단계별 전환 건수 비율 |

### ERP 없이 Excel로 시작하기

경영 지표는 **별도 시트가 필요 없습니다.** 위 부서(영업/구매/물류/생산/재무)의 데이터를 조합하면 됩니다.

> **NOAH 대시보드의 Order Book 페이지가 이 구조입니다.** SO(수주) + DN(출고)만으로 워터폴, BtB, Backlog Cover, Aging을 전부 만들었고, PO(발주)를 추가해서 마진/커버리지까지 확장했습니다.

```sql
-- Order Book 워터폴 (개념 — 실제 NOAH 쿼리는 sql/order_book.sql 참조)
-- 월별 Input(수주) / Output(출고) → 월별 합산 후 Rolling Ending
WITH monthly AS (
    SELECT Period,
           SUM(Value_Input_amount) AS Input,
           SUM(Value_Output_amount) AS Output
    FROM order_book
    GROUP BY Period
)
SELECT Period, Input, Output,
       SUM(Input - Output) OVER (ORDER BY Period) AS Ending
FROM monthly

-- Book-to-Bill
SELECT Period,
       SUM(Value_Input_amount) / NULLIF(SUM(Value_Output_amount), 0) AS BtB
FROM order_book GROUP BY Period

-- 전환 퍼널 (SO → PO → DN)
SELECT
  (SELECT COUNT(DISTINCT SO_ID) FROM 수주대장) AS SO건수,
  (SELECT COUNT(DISTINCT SO_ID) FROM 발주대장) AS PO건수,
  (SELECT COUNT(DISTINCT SO_ID) FROM 출고대장) AS DN건수
```

핵심: **각 부서가 자기 시트를 잘 관리하면, 경영 대시보드는 자동으로 나옵니다.**

## 9. 설계 순서 (ERP 없이 시작할 때)

1. **마스터부터** — 고객, 품목, 계정과목 목록 정리
2. **핵심 트랜잭션** — 수주 → 발주 → 출고 (이것만으로 영업/구매/물류 커버)
3. **회계** — TB만 있으면 P&L/BS 가능
4. **파생 지표** — Order Book, 마진, BtB는 위 데이터 조합

### NOAH 프로젝트가 따른 순서

```
1단계: Excel에 SO/PO/DN 시트 설계 (마스터+트랜잭션 한 파일)
2단계: sync_db.py로 SQLite 동기화
3단계: dashboard.py로 시각화
4단계: 무설치 배포 (Portable)
→ ERP 없이도 동일한 분석 체계 구축 완료
```
